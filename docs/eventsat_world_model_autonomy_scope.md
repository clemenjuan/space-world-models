# EventSat World-Model Autonomy Scope

This note captures the intended direction for using learned world models in the
EventSat/autonomous-operations experiments.

## Core Idea

The learned world model should support two related but distinct uses:

1. Fast evaluation of proposed operations.
2. Action selection through a planner/controller built on top of the model.

The model itself is not an autonomous operator. It predicts what is likely to
happen if a sequence of operational actions is applied. Autonomy appears when a
controller uses those predictions to choose actions.

## Track 1: World Model As Fast Evaluator

Use the world model as a cheap inner-loop simulator for candidate action
sequences produced by other agents.

Candidate action sources can include:

- scripted heuristic operators;
- AUTOPS rule-based EventSat agents;
- AUTOPS schedule-based ground planners;
- RL policies;
- LLM/HLLM agents;
- human-designed schedules.

Workflow:

```text
candidate actions -> world model rollout -> predicted resources/data/risk
                  -> score/rank/filter -> optional high-fidelity validation
```

The value here is speed. Instead of running every candidate through the full
high-fidelity AUTOPS simulation, the world model can triage many plans quickly.
The high-fidelity simulator remains the source of truth for selected candidates,
benchmarks, and safety validation.

This is useful even before the world model chooses actions by itself.

## Track 2: World Model Plus Planner/Controller

To make the system choose actions, wrap the world model in a planner/controller.
The likely first version is model predictive control (MPC):

```text
current history
  -> sample candidate mode sequences
  -> predict outcomes with the world model
  -> decode/score mission and resource metrics
  -> execute the first action
  -> observe the real next state
  -> repeat
```

The planner can be simple at first:

- enumerate or sample short horizons over the 7 EventSat modes;
- use action masks for impossible or unsafe modes;
- score predicted outcomes with mission reward, downlink, observation time,
  battery safety, storage safety, and anomaly risk;
- execute only the first selected action to keep feedback closed-loop.

This creates a new autonomous operations baseline:

```text
LeWM dynamics model + planner/controller
```

That baseline can then be compared against AUTOPS symbolic, RL, LLM, HLLM, and
hybrid agents.

## What The Current Model Does

The current EventSat LeWM/VectorJEPA model learns action-conditioned dynamics:

```text
recent observations + one-hot mode actions -> next latent observation embedding
```

It does not currently learn a policy. In the first week simulation, actions were
generated by a scripted heuristic operator, not by the world model.

The current model is sufficient for:

- measuring latent prediction quality;
- testing whether dynamics are learnable;
- evaluating known action traces in the world-model latent space;
- bootstrapping a fast evaluator prototype.

It is not yet sufficient for robust autonomous action choice, because the
planner needs interpretable predicted quantities such as battery, storage,
ground-pass state, data backlog, reward, and safety risk.

## Required Additions

To use the world model for action selection, add:

- state/reward decoders from latent embeddings to operational metrics;
- a planner/controller loop that proposes, rolls out, scores, and executes
  actions;
- action masks/safety guards for communication outside contact, unsafe battery
  use, storage overflow, anomaly handling, and invalid mode choices;
- training data with broader action coverage, including RL/LLM/scheduler
  actions, not only the scripted heuristic;
- high-fidelity validation in AUTOPS for selected policies/plans.

## Using RL/LLM Actions With The World Model

RL or LLM agents can propose actions or schedules, and the world model can
evaluate them before running the full simulation.

This creates a useful architecture:

```text
RL/LLM/scheduler proposal
  -> world-model rollout and risk scoring
  -> reject, repair, rank, or approve
  -> high-fidelity AUTOPS execution for selected plans
```

This is especially attractive for LLM/HLLM operations agents, where a fast
predictive checker can catch plans that look plausible linguistically but are
resource-unsafe or operationally ineffective.

## AUTOPS Integration Scope

The AUTOPS repo should be used as the high-fidelity environment and benchmark
suite. The world model can be integrated in stages:

1. Export AUTOPS EventSat trajectories into the local world-model dataset
   schema: observations, actions, raw state, rewards, and metrics.
2. Evaluate the current toy-trained model out-of-distribution on AUTOPS
   trajectories.
3. Train or fine-tune on AUTOPS-generated trajectories.
4. Add decoders for AUTOPS-relevant state and mission metrics.
5. Implement a LeWM planner/controller as an AUTOPS representation/agent.
6. Compare against existing AUTOPS EventSat agents: rule-based, schedule-based,
   RL, LLM, HLLM, and hybrid variants.

The goal is not propulsion or orbital trajectory optimization. The goal is
autonomous spacecraft operations: mode selection, payload pipeline management,
communication timing, data handling, resource safety, and anomaly response.

## Near-Term Experiments

1. Fast evaluator prototype.
   Run scripted, RL, LLM, and schedule-based action traces through the world
   model and compare predicted vs true AUTOPS outcomes.

2. Decoder experiment.
   Train heads for battery SoC, OBC/Jetson data, downlink, backlog counters,
   contact flags, and reward.

3. LeWM-MPC toy controller.
   Use the simplified EventSat environment to test whether a planner over the
   world model can beat the scripted heuristic.

4. LeWM-MPC in AUTOPS.
   Port the controller into AUTOPS and compare it against the existing
   autonomous operations paradigms.

## Success Criteria

The direction is promising if the world model can:

- rank candidate action sequences similarly to the high-fidelity simulator;
- reject unsafe or ineffective RL/LLM plans before full simulation;
- reduce the number of expensive AUTOPS evaluations needed;
- support a planner/controller that performs competitively with existing
  symbolic/RL/LLM agents;
- expose where model uncertainty or out-of-distribution actions require
  fallback to high-fidelity simulation or conservative rules.
