# FDIR LeWM Experiment Alignment

This note explains the current fault-detection experiment and how to read the
local FDIR results board.

## Problem

We simulate spacecraft telemetry under nominal coupled linear dynamics, then
train a LeWM model only on nominal episodes. At evaluation time we inject a fault
at the hidden-state dynamics level and look for a rise in latent prediction
error.

The hidden state and observation both have eight telemetry channels:

```text
[solar_array_voltage, battery_soc, panel_temp, obc_temp,
  rw_speed_x, rw_speed_y, rw_speed_z, bus_current]
```

The environment stores the true state for diagnostics. The model receives the
noisy observation stream plus a one-hot action. In this step the action is always
nominal and does not affect dynamics; it is plumbed so the same LeWM interface
can later support recovery policies.

## Why Train On Nominal Only?

FDIR detection is framed as a novelty problem. The model learns how nominal
telemetry evolves:

```text
nominal telemetry history + nominal action history -> predicted next latent
```

At fault time, the observation stream moves away from the learned nominal
manifold. We measure that as surprise:

```text
surprise_t = || z_hat_t - enc(o_t) ||^2
```

where `z_hat_t` is the predictor output from the previous latent history, and
`enc(o_t)` is the encoder output for the actual observation.

## Fault Model

Faults are injected into the state evolution, not merely added to the emitted
observation. This matters because coupled telemetry channels can respond to a
faulted channel. The environment supports:

- `stuck_at`: latch one state channel at its onset value;
- `drift`: add a per-step ramp to one channel;
- `spike`: add a transient state-level impulse for a configured duration.

The small deterministic smoke test uses a 20-step solar-array-voltage spike
because it reliably clears the relative `post > pre + 3 * std(pre)` threshold in
seconds. Longer training runs can inspect all three fault modes on the board.

## What The Current LeWM Learns

The FDIR-JEPA path is the same model family as OD, with new dimensions:

```text
8D telemetry -> MLP encoder -> 192D latent
4D one-hot action -> action encoder -> 192D action latent
latent history + action latent -> autoregressive predictor -> next 192D latent
```

The loss is unchanged from OD:

```text
prediction MSE + 0.09 * SIGReg
```

No fault labels, recovery decisions, or auxiliary anomaly losses are used.

## Board Interpretation

The FDIR board compares:

- raw nominal telemetry from the generated dataset;
- a deterministic faulted rollout from the same simulator;
- LeWM latent surprise before and after the fault;
- a simple observation z-score energy baseline;
- training curves from the newest FDIR run under `~/.cache/stable-pretraining`.

The baseline is intentionally simple. It is not a flight-quality detector; it is
a sanity reference that asks whether raw normalized telemetry energy also rises.
The LeWM detector is more interesting when faults are subtle, coupled, temporal,
or close to nominal marginal ranges but inconsistent with learned dynamics.

## Reference Hyperparameters

The original LeWM website and code describe a larger pixel-based setup than this
CPU-friendly telemetry experiment. Their public training config uses 100 epochs,
GPU acceleration, bf16 precision, batch size 128, image size 224, embed dim 192,
history size 3, one prediction step, AdamW with lr 5e-5 and weight decay 1e-3,
and SIGReg weight 0.09 with 17 knots and 1024 projections. Their model config
uses a ViT-tiny pixel encoder, a 6-layer predictor with 16 heads, MLP dim 2048,
and BatchNorm MLP projector heads.

Our OD/FDIR telemetry adaptation keeps the shared LeWM training objective and
key latent values (`embed_dim=192`, `history_size=3`, `num_preds=1`, `SIGReg=0.09`)
but swaps the pixel encoder for a vector MLP, uses smaller predictor/projector
heads, CPU precision 32, and 16 epochs for the real-scale local run so it matches
the OD experiment convention.


## Improved Evaluation Protocol

The board now separates the illustrative single fault trace from a calibrated
multi-scenario benchmark. The benchmark artifact is
`data/figures/fdir_detection_benchmark.json`, generated by:

```bash
.venv/bin/python scripts/evaluate_fdir_detection.py
```

The current protocol calibrates LeWM surprise and observation z-score thresholds
on nominal training episodes, using:

```text
threshold = max(mean + 3 * std, 99.5th percentile)
alarm = 3 consecutive above-threshold samples
detection = first alarm within 60 steps after the fault onset
```

Each fault scenario is evaluated over 16 simulator seeds with the same nominal
action stream. The table reports detection rate, median detection delay,
pre-fault alarm rate, and post/pre score lift. This makes the experiment less
anecdotal than the original one-spike plot.

## Current Benchmark Readout

For the 16-epoch run `fdir-lewm-192x256-16ep` (`8ba729c21125`), the calibrated
benchmark says:

- LeWM has no pre-fault alarms in the evaluated seeds and detects both voltage
  spike scenarios at 100%.
- LeWM spike detection is delayed: median 26-27 steps after onset.
- LeWM misses the evaluated slow/fast voltage drifts, voltage stuck-at,
  wheel-speed drift, wheel stuck-at, and panel-temperature drift under this
  threshold.
- The raw z-score baseline detects the spike and drift scenarios at 100%, with
  shorter median delays of 2-14 steps.
- Both detectors are weak on stuck-at faults in this environment, because the
  latched value often remains close to the nominal marginal range.

The conclusion is therefore deliberately modest: the current FDIR LeWM run is a
working novelty detector for abrupt impulse faults, but it does not yet show a
LeWM advantage over a simple marginal z-score detector. The next useful
experiment should target temporal or coupled faults that are marginally
plausible but dynamically inconsistent, or add a learned state/observation
residual probe so the latent surprise can be interpreted at channel level.

## Current Limits

This is detection only. There is no recovery policy, no fault classification
head, no supervised fault labels, and no calibrated probability of anomaly yet.
The current comparison now has nominal threshold calibration, detection delay,
and per-fault-mode robustness checks. Future work should add held-out nominal
calibration splits, precision/recall curves across threshold sweeps, fault
classification, recovery actions, and channel-level residual explanations.
